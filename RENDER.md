# Deploying to Render — zero-touch

This repo ships a [Blueprint](./render.yaml) that wires everything automatically:

- **kitchen-ecom-api** — Node web service (Express + Socket.IO) with a 1 GB persistent disk at `/var/data/uploads`.
- **kitchen-ecom-web** — Static site for the Vite React PWA.
- **kitchen-ecom-secrets** — Shared env group; you fill it in once.

What's automatic now:

- Cross-service URLs are derived via `fromService`. The web service gets the API's hostname into `VITE_API_HOST`, and the API gets the web's hostname into `CLIENT_URL`. The server prepends `https://` automatically when it sees a bare host.
- The API's own `PUBLIC_URL` falls back to Render's auto-injected `RENDER_EXTERNAL_URL`. Nothing to set.
- JWT secrets are generated by Render (`generateValue: true`).
- Roles, super-admin, sample categories, and the storefront setting are seeded on every API boot (idempotent — `AUTO_SEED=true`).

MongoDB still lives outside Render. Use [Atlas](https://www.mongodb.com/atlas) — the free M0 tier is enough.

## One-time setup (≈ 5 minutes)

### 1. Atlas

1. Atlas → new project → **M0 Free** cluster.
2. Database Access → add a user (note the password).
3. Network Access → **Allow access from anywhere** (`0.0.0.0/0`) — Render outbound IPs aren't fixed on lower plans.
4. Cluster → Connect → Drivers → copy the `mongodb+srv://…` string. Replace `<password>` and append the DB name, e.g. `/kitchen_ecom?retryWrites=true&w=majority`.

### 2. Apply the Blueprint

1. [Render Dashboard](https://dashboard.render.com) → **New +** → **Blueprint** → pick this repo.
2. Render reads `render.yaml`. It surfaces the secret keys grouped under `kitchen-ecom-secrets` and asks you to fill them.

### 3. Fill the env group (only thing you type)

In the prompt during Blueprint apply (or later: **Env Groups → kitchen-ecom-secrets**):

| Key | Value |
| --- | --- |
| `MONGO_URI` | the Atlas connection string from step 1 |
| `SEED_SUPER_ADMIN_PHONE` | e.g. `+233500000000` (E.164 with `+`) |
| `SEED_SUPER_ADMIN_PASSWORD` | a strong password — your login |
| `ARKESEL_API_KEY` | leave blank for stub mode (OTPs printed to logs) |
| `PAYSTACK_SECRET_KEY` | leave blank for stub mode (auto-success checkout) |
| `PAYSTACK_PUBLIC_KEY` | leave blank for stub mode |
| `PAYSTACK_WEBHOOK_SECRET` | leave blank for stub mode |
| `VITE_PAYSTACK_PUBLIC_KEY` | same as above, blank for stub |
| `VITE_WHATSAPP_NUMBER` | E.164 digits only, no `+`, e.g. `233240000000`. Blank if unused. |

Save. Both services build. The API will:

- Connect to Atlas.
- Run the seed (creates roles + super admin + 5 sample categories + storefront setting).
- Start listening, with `CLIENT_URL` pointing at the static site and `PUBLIC_URL` self-discovered.

The static site picks up `VITE_API_HOST` at build time and constructs `https://<api-host>/api` for axios and Socket.IO.

### 4. (Optional) Paystack webhook

If you ever fill in live Paystack keys, set the webhook in Paystack Dashboard → Settings → API Keys & Webhooks:

- URL: `https://<api-host>/api/payments/webhook/paystack`
- Use the same secret you put in `PAYSTACK_WEBHOOK_SECRET`.

## Smoke test

1. Open the static site URL. Home loads with the 5 seeded categories.
2. Sign in as super admin with `SEED_SUPER_ADMIN_PHONE` / `SEED_SUPER_ADMIN_PASSWORD`.
3. Register a separate test account. In stub mode the OTP appears in the **kitchen-ecom-api** logs; in live Arkesel mode it arrives via SMS.
4. Upload a product image in admin. The URL should be `https://<api-host>/uploads/…`.
5. Place a test order. Stub mode → `/checkout/stub-pay`. Live Paystack → real checkout.

## What lives where

| Concern | Location |
| --- | --- |
| HTML/JS/CSS bundle | static site CDN (kitchen-ecom-web) |
| API + WebSocket | web service (kitchen-ecom-api) |
| MongoDB | Atlas (external) |
| Product images | Render disk at `/var/data/uploads` |
| Refresh tokens, sessions | MongoDB (no Redis needed) |
| Shared secrets | Render env group `kitchen-ecom-secrets` |

The disk survives deploys and restarts. It's tied to the API service; deleting the service deletes the disk.

## Custom domains

1. **kitchen-ecom-web → Settings → Custom Domains** → add e.g. `shop.example.com`.
2. **kitchen-ecom-api → Settings → Custom Domains** → add `api.example.com`.

The Blueprint cross-refs use `host` from each service. Adding a custom domain doesn't change the auto-injected `host` (it stays on the `.onrender.com` name). If you want the API to advertise the custom API domain in image URLs, override on the API service:

| Key | Value |
| --- | --- |
| `PUBLIC_URL` | `https://api.example.com` |
| `CLIENT_URL` | `https://shop.example.com,https://kitchen-ecom-web.onrender.com` (comma-separated to keep both working) |

For the static site, override:

| Key | Value |
| --- | --- |
| `VITE_API_URL` | `https://api.example.com/api` |
| `VITE_SOCKET_URL` | `https://api.example.com` |

Redeploy the static site so Vite picks them up (Vite bakes envs at build time). The presence of `VITE_API_URL` / `VITE_SOCKET_URL` overrides the auto-derived `VITE_API_HOST` path.

- API service: Starter $7/mo (Starter is required to attach a disk).
- Disk: 1 GB ≈ $0.25/mo.
- Static site: free.
- MongoDB Atlas M0: free.
- **Total: ~$7–8/mo.**

To cut cost: switch uploads to Cloudinary or drop to ephemeral storage (changes in `server/src/routes/upload.routes.js`).

## Re-running the seed manually

Auto-seed runs on every boot. If you want to invoke it by hand (e.g. after wiping a category), open **kitchen-ecom-api → Shell** and run:

```
npm run seed
```

To disable auto-seed entirely, set `AUTO_SEED=false` on the API service.

## Troubleshooting

| Symptom | Likely cause |
| --- | --- |
| Static site: `Network Error` | `VITE_API_HOST` empty — env group not applied or static site didn't rebuild after group save. Trigger a manual deploy. |
| API logs: `CORS: origin … not allowed` | The static site URL didn't match `CLIENT_URL`. Check the `fromService` reference resolved — it shows under the API service's Environment tab as a `[from kitchen-ecom-web]` chip. |
| API logs: `Missing required env var: MONGO_URI` | Env group didn't get linked. Confirm `kitchen-ecom-secrets` is bound to the API service under Environment. |
| Uploaded images return 404 | `RENDER_EXTERNAL_URL` not seen by the service — happens if you set `PUBLIC_URL` manually to a wrong value. Clear it to fall back. |
| Socket.IO won't connect | `VITE_SOCKET_URL` was set manually with a trailing `/api`. It must be the root host. Clear it to fall back to `VITE_API_HOST`. |
| Login works but refresh fails | Clock skew or rotated `JWT_REFRESH_SECRET`. Regenerate (delete the env var, redeploy — `generateValue` recreates it). |
